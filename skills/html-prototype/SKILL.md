# HTML Prototype Skill

## 目标

在 propose → plan 之间插入**可视化确认**环节：将需求规格转化为可在浏览器中直接打开的静态 HTML 原型，让用户通过视觉和交互确认功能设计，消除文本描述的理解偏差。

## 核心原则

1. **零依赖** — 纯 HTML + CSS + 内联 JS，无需安装任何工具，直接浏览器打开
2. **快速迭代** — 原型生成速度优先于视觉完美度
3. **交互可感知** — 关键交互用 JS 模拟（点击、切换、表单验证提示），让用户"感受"流程
4. **一次性产物** — 原型是确认工具，不是最终代码；确认后进入正式开发阶段

## 执行步骤

### Step 1: 读取需求规格

从 `specs/active/` 读取 propose 阶段产出的需求文档，提取：

| 提取项 | 说明 |
|--------|------|
| 页面列表 | 需要展示哪些页面/视图 |
| 核心交互 | 按钮点击、表单提交、导航跳转、状态切换 |
| 数据展示 | 列表、表格、卡片、详情等数据呈现方式 |
| 用户流程 | 关键操作路径（如：登录 → 列表 → 详情 → 编辑 → 提交） |
| 约束条件 | 终端类型（PC/移动端）、风格偏好 |

### Step 2: 生成原型结构

创建 `prototype/{feature}/` 目录：

```
prototype/{feature}/
├── index.html          # 入口页（含导航到各页面）
├── pages/
│   ├── page-1.html     # 各功能页面
│   ├── page-2.html
│   └── ...
└── README.md           # 原型说明（如何查看、反馈方式）
```

### Step 3: 编写 HTML 原型

**技术要求：**

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{页面标题} - 原型预览</title>
  <style>
    /* 内联 CSS，不依赖外部文件 */
    /* 使用系统字体栈 */
    /* 响应式基础布局 */
  </style>
</head>
<body>
  <!-- 顶部提示条：这是原型 -->
  <div class="prototype-banner">
    ⚠️ 这是功能原型，用于确认交互和布局，非最终视觉效果
  </div>

  <!-- 页面内容 -->

  <script>
    // 内联 JS：模拟交互
  </script>
</body>
</html>
```

**视觉风格：**

- 使用简洁的线框风格，重功能布局而非视觉装饰
- 颜色：主色 `#2563eb`、成功 `#16a34a`、警告 `#f59e0b`、错误 `#dc2626`、背景 `#f8fafc`
- 圆角、阴影适度，体现层级关系
- 表单控件使用浏览器原生样式 + 轻微美化

**交互模拟：**

| 交互类型 | 模拟方式 |
|----------|---------|
| 页面跳转 | 真实 `<a href>` 链接到其他 HTML 文件 |
| Tab 切换 | JS 控制显示/隐藏对应内容区 |
| 表单提交 | 点击后显示成功提示，不真实发送 |
| 列表操作 | 使用 Mock 数据填充，删除/编辑用弹窗模拟 |
| 弹窗/抽屉 | JS 控制 display 切换 |
| 加载状态 | 点击后短暂显示 loading 再展示结果 |
| 表单验证 | 简单的必填 + 格式校验提示 |

### Step 4: 生成原型说明

创建 `prototype/{feature}/README.md`：

```markdown
# {Feature} 原型说明

## 查看方式

直接在浏览器中打开 `index.html` 即可预览。

## 页面清单

| 页面 | 文件 | 功能说明 |
|------|------|---------|
| 首页/入口 | index.html | {说明} |
| {页面名} | pages/{name}.html | {说明} |

## 可交互的元素

- {列出哪些按钮/表单/导航可以点击操作}

## 确认要点

请重点确认以下内容：
1. 页面布局是否符合预期？
2. 操作流程是否顺畅？
3. 信息展示是否完整？
4. 是否有遗漏的功能点？

## 反馈方式

确认后告知 AI：
- ✅ "原型确认通过" → 进入 plan 阶段
- ❌ "需要修改：{具体修改点}" → 重新生成
```

### Step 5: 用户确认与需求联动

将原型路径告知用户，等待确认：

```
✅ 原型已生成：prototype/{feature}/index.html
   请在浏览器中打开查看，确认后我们进入下一步。

   确认选项：
   - 通过 → 进入 /forge-plan
   - 需修改 → 描述修改点（自动判断是否涉及需求变更）
   - 放弃 → 回到 /forge-propose 重新梳理需求
```

**修改分类与联动逻辑：**

收到用户修改反馈后，自动判断修改类型并执行对应动作：

| 修改类型 | 判断依据 | 动作 |
|----------|---------|------|
| **布局调整** | 位置移动、样式变化、排版优化 | 只重新生成 HTML，spec 不变 |
| **功能变更** | 增减功能点、流程步骤变化、字段增删、交互逻辑改变 | 重新生成 HTML + 同步更新 spec |
| **重大偏离** | 目标用户变化、核心流程推翻、技术方案不可行 | 回退到 /forge-propose |

**功能变更时的 Spec 同步流程：**

```
用户反馈: "加一个批量选择功能" / "导出前需要预览"
    │
    ▼
Step 5a: 判定为功能变更
    │
    ▼
Step 5b: 更新 specs/active/{slug}/proposal.md
    ├── 在 spec 中追加/修改对应功能描述
    ├── 标记变更来源: "原型评审反馈"
    └── 记录变更内容摘要
    │
    ▼
Step 5c: 重新生成 HTML 原型
    └── 基于更新后的 spec 重新生成
    │
    ▼
Step 5d: 再次请用户确认
```

**Spec 变更标记格式：**

在 `specs/active/{slug}/proposal.md` 中追加：

```markdown
## 原型评审变更记录

| 轮次 | 变更内容 | 类型 | 日期 |
|------|---------|------|------|
| #1 | 增加批量选择功能 | 功能新增 | {date} |
| #2 | 导出前增加数据预览步骤 | 流程变更 | {date} |
```

**断路器：** 功能变更累计超过 3 轮 → 暂停，建议回到 propose 重新梳理需求范围（可能原始需求理解有偏差）。

## 适用场景判断

| 场景 | 是否生成原型 |
|------|-------------|
| 涉及新页面/新视图 | ✅ 建议 |
| 涉及交互流程变更 | ✅ 建议 |
| 涉及表单/列表/数据展示 | ✅ 建议 |
| 纯 API / 后端逻辑 | ❌ 跳过 |
| 修复 UI bug（已知问题） | ❌ 跳过 |
| 简单样式调整 | ❌ 跳过 |

## 原型质量标准

- [ ] 所有页面可通过浏览器直接打开（双击即可）
- [ ] 页面间导航链接正确
- [ ] 关键交互有响应（点击有反馈）
- [ ] Mock 数据贴近真实业务
- [ ] 移动端/PC 端基础适配（如需求指定）
- [ ] 无外部依赖（不引用 CDN、不需要服务器）

## 与其他 Skill 的关系

```
propose (澄清需求，产出 spec)
  │
  ▼
html-prototype (可视化确认) ◄── 本 Skill
  │
  ├── 确认通过 → plan (任务拆解)
  ├── 需修改 → 重新生成原型
  └── 放弃 → 回到 propose
  
plan → execute → review → verify → ship
```

## 不做的事

- 不生成可部署的前端代码（那是 execute 的职责）
- 不引入任何框架或构建工具
- 不追求像素级完美（原型重确认，不重美观）
- 不替代正式的 UI 设计（如果项目有设计师参与）
- 不保留原型代码到最终产品中
